## 這套制度是什麼

`governance/` 底下六份檔+重寫過的 CLAUDE.md。核心思想只有一句:
**把「哪些判斷可以便宜地做」和「哪些判斷不能省」分開,前者寫成規則照做,後者才動用貴模型或使用者。**

閱讀順序(新 session 第一次進來):CLAUDE.md(已常駐)→ 本信 → 用到什麼讀什麼(路由表在 CLAUDE.md)。不要一口氣全讀,那正是 diagnosis.md 一-2 要防的行為。

## 三件使用者沒問、但我認為最重要的事

### 1. Markdown 規則是「建議」,hooks 才是「強制」——關鍵規則被反覆略過時,升級成 hook

這套制度全部靠「模型讀了照做」。弱模型在長 session 後半段會忘記或跳過(context 稀釋是物理現象,不是態度問題)。如果發現某條硬規則(例如「改檔先備份」)被反覆違反,正確的修法不是把規則寫得更兇,而是請使用者用 `/update-config` 把它變成 settings.json 的 hook——由 harness 強制執行,模型想忘也忘不掉。判斷基準:同一條規則第三次被略過,就該提議 hook 化。

### 2. 這個 repo 的真正資產是 knowledge/(300+ 檔),制度只是保護它的殼

维運 KB 的成本大頭在「找」不在「寫」。@PROJECT 的 source-codex 已有 index.md 層,善用它:**永遠先讀 index 再深入**,而不是 grep 全庫。如果未來新增專案 KB,第一件事是建 index.md(每目錄一份 ≤50 行的地圖),否則弱模型的搜尋成本會隨檔案數線性上升。KB 的新鮮度(index 是否還對)比 KB 的完整度更值得花額度。

### 3. 制度只會透過「使用者糾正 → 寫回」變好,不會自己變好

Rita 是單人與 Claude Code 長期協作。每次她糾正你的做法,那就是最高價值的訓練訊號——當場問一句「這條要不要進 lessons.md」,四行寫完(格式在 maintenance-protocol.md §3)。已有先例:memory 裡的 `feedback_narrate-then-act` 就是一次糾正沉澱成的規則,後來進了 CLAUDE.md 硬規則 3。這條回路是制度唯一的進化機制,斷了它,制度就凍結在 2026-07-05 的水準。

## 誠實條款:哪些產出我信心最低、為什麼

按信心由低到高:

1. **model-dispatch.md §1 的「任務型別 → model/effort」對應表**:這是先驗判斷,沒有在此環境實測校準過。例如「彙整用 haiku」可能在複雜 KB 摘要上品質不足。用法:當成起點,錯了就按 §5 升級,並把校正結果記 lessons.md。
2. **升降級路徑的具體數字**(錯一次升級、兩次帶軌跡、最多兩輪):合理的策略參數,但「兩」不是實驗得出的,是權衡「重試成本 vs 升級成本」的估計。可以調,調了記錄。
3. **diagnosis.md 的「前三名」排序**:基於單一 session 的觀察。特別是「claude-api skill 誤觸發」的嚴重度取決於 Rita 未來是否常寫 Anthropic API 程式碼——如果常寫,該條要降級。
4. **「主線 Agent 工具呼叫沒有 effort 參數」**:2026-07-05 的工具 schema 如此,但這類介面迭代快,可能很快過時。過時的表現:官方文件出現該參數 → 直接更新 model-dispatch.md §0(屬可自行修改)。
5. **「被導向 Opus 4.8 的請求是否消耗額度」:完全未查證**,只能標註。建議 Rita 有空時在 usage 儀表板實測一次,把結論寫進 model-dispatch.md §0。

還有一句要說白:**拆解、驗證、多樣本評審補得了執行品質;模糊題與品味判斷補不了。**「這份 ADR 寫得好不好」「這個抽象值不值得」這類問題,弱模型照 checklist 也判不準——遇到就升級模型或直接問 Rita,不要假裝 rubric 能覆蓋一切。這正是把日常執行省下來的額度該花的地方。
