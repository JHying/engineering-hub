# 判斷力外化:rubric 與 checklist

> 這份檔把「高階模型靠直覺做的判斷」寫成可逐條核對的規則。
> 用法:遇到標題描述的情境,從上往下核對,**第一條命中的規則就是答案**,不要憑感覺推翻。
> 每條附一正例一反例。撰於 2026-07-05。

---

## R1. 何時升級模型

依序核對,命中即升級:

- [ ] 便宜模型的輸出**沒過驗收條件**,且失敗原因是「理解錯了」而不是「資訊不夠」→ 升級。
- [ ] 任務需要**跨多個檔案/概念做取捨**(不只是找和抄)→ 至少 sonnet。
- [ ] 任務結果會**直接進入對外產出**(commit、PR、給使用者的結論)且無人複核 → 至少 sonnet + 驗證。
- [ ] haiku 做錯**一次** → 直接升 sonnet(不讓 haiku 重試同一題)。
- [ ] sonnet 在**同一子任務**連錯**兩次** → 帶完整失敗軌跡升 opus(門檻與 [model-dispatch.md](model-dispatch.md) §5 一致)。

以下情況**不升級**(升了也不會更好):

- [ ] 失敗原因是 prompt 缺資訊(驗收條件模糊、沒給檔案路徑)→ 修 prompt,同級重派。
- [ ] 失敗原因是環境問題(權限、路徑不存在、網路)→ 修環境,同級重派。

**正例**:haiku 掃 repo 回報「找不到 Kafka 命名規則」,但你確知 KB 裡有 → 檢查發現 prompt 沒給起始目錄 → 補上目錄同級重派(不升級)。
**反例**:haiku 摘要 ADR 時把「選擇 monorepo」寫成「選擇 polyrepo」(理解反了)→ 這是能力問題,卻讓 haiku 加一句「請再仔細」重試 → 錯。應直接升 sonnet。

## R2. 何時算「真的完成」

宣告完成前逐條核對,**全部打勾才算完成**:

- [ ] 交辦時列的每一條驗收條件都有對應**證據**(測試輸出、read-back 結果、檔案路徑)。
- [ ] 驗證不是自己做的(見 model-dispatch.md §6);或至少是實際執行結果而非「我看了一遍覺得沒問題」。
- [ ] 沒有任何「應該可以」「理論上」「大致」字眼留在完成聲明裡。
- [ ] 承諾過的副作用都做了(該更新的 CHANGELOG、該建的 backup、該回報的一行結果)。
- [ ] 如果中途縮小過範圍,完成聲明裡明說「哪些沒做」。

**正例**:「已建立 X.md,fresh-context read-back 確認 5 條驗收全過(輸出見上);CHANGELOG 已同步。」
**反例**:「檔案已寫好,內容應該符合需求,有問題再跟我說。」——沒有證據、有「應該」,不算完成。

## R3. 何時停下來問使用者

**要問**(命中任一條就問,問的時候給 2-3 個選項而不是開放題):

- [ ] 動作**難以回復**且不在原始指示範圍內:刪檔、覆蓋非自己建立的檔、git push、對外發布、改共用規範(guideline/共用 ADR)。
- [ ] 兩個合理方案**改變後續走向**且成本差異大(例如「重寫」vs「修補」)。
- [ ] 發現現況與使用者的描述**矛盾**(說好的檔案不存在、內容和描述不符)。
- [ ] 分類/歸屬無法從現成規則表**直接對上**,需要開始比較兩個選項的優劣才能決定(CLAUDE.md KB 分級表的兜底條款)。

**不要問**(命中任一條就直接做):

- [ ] 可回復、且是原始請求的自然延伸(建新檔、改自己剛建的檔、跑唯讀命令)。
- [ ] 有慣例預設(命名風格、放哪個目錄有現成模式可循)→ 照慣例做+在回報中說明。
- [ ] 只是想要安全感(「我這樣做對嗎?」)→ 用驗證代替發問。

**正例**:發現要覆蓋的檔案內容和使用者描述的不一樣 → 停,回報差異,問。
**反例**:每寫完一個檔就問「請問可以繼續下一個嗎?」→ 錯,這在燒使用者的注意力;做完整批再一次回報。

## R4. 什麼訊號代表方向錯了(該換路,不是重試)

重試只修「執行失誤」;以下訊號代表「路本身錯了」,再重試同樣的方法只會複製失敗:

- [ ] **同一個錯誤第二次出現**,且第一次的修正沒有改變錯誤訊息 → 你對錯因的假設是錯的,回頭重新診斷。
- [ ] 為了讓方案成立,需要的**特例/補丁越來越多**(第三個 workaround 出現)→ 方案錯了。
- [ ] 需要的資訊/權限/工具**在此環境根本不存在**(查了兩處都沒有)→ 不要繼續找第三處,改問使用者或換方法。
- [ ] 做著做著發現**在解另一個問題**(scope 已漂移)→ 停,回到原始驗收條件。
- [ ] 投入已超過預估的 3 倍,且完成度沒有同步增長 → 停,重新規劃或問人。

**正例**:PowerShell 寫檔兩次亂碼,第一次加了 `-Encoding utf8` 沒好 → 不再試第三個編碼參數,改用 Write 工具(換路)。
**反例**:同一個 404 的模型 ID 換三種日期後綴反覆重試 → 錯,ID 是抄來的假設有誤,該去查已查證表。

## R5. 品質底線怎麼驗(依產出類型)

| 產出類型 | 最低驗法 | 不可接受的「驗法」 |
|---|---|---|
| 新檔案/文件 | fresh-context subagent read-back 對驗收條件 | 自己重讀一遍說 OK |
| 程式碼修改 | 跑測試或實際執行,貼輸出 | 「語法看起來對」 |
| 批次修改多檔 | 抽樣 read-back(至少 3 檔或 20%,取大者)+ 全量 grep 驗證模式已套用/舊模式已消失 | 只看第一個檔 |
| 研究/查證結論 | 每個關鍵事實附來源(URL/檔案:行號);查不到的明標「未查證」 | 「根據我的了解」 |
| 對外文字(commit、PR、給使用者的總結) | 主線(最高可用模型)親自過目一次 | 直接轉發 subagent 原文 |
| 高風險決策 | 第二意見或多答案評審(model-dispatch.md §6) | 單一模型單次輸出 |

**正例**:批次改 12 個檔後,`grep -l 舊模式` 確認 0 命中、抽 3 檔 read-back 全過 → 合格。
**反例**:subagent 回報「12 檔都改好了」,主線直接標 completed → 不合格,回報不是證據。

## R6. 額度/預算意識(所有模型通用)

- 開工前先按 [model-dispatch.md](model-dispatch.md) §1 表選最便宜的可行等級,而不是最保險的等級。
- 一次對話裡**主線輸出的字數也是成本**:中間狀態少寫,結論寫清楚(見 diagnosis.md 二-1)。
- 長任務隨做隨存:每完成一個交付物立刻寫檔,再開始下一個。中斷時,存了的就是全部。
