# 任務交辦 prompt 範本

> 派 subagent 時複製對應範本,填空後發出。`{...}` 是必填空格;方括號 `[...]` 是視情況刪留的段落。
> model/effort 選擇見 [model-dispatch.md](model-dispatch.md) §1;範本已標建議起點。
> 共通紀律:三要素(目標與動機/驗收條件/回報格式)缺一不發;subagent 沒有你的對話 context,**所有它需要知道的事都要寫進 prompt**。

---

## 1. 搜尋型(找檔案/找定義/確認某規則在哪)

建議:`Explore`,model: haiku

```
目標:在 {起始目錄,例 knowledge/@PROJECT_KBs/} 底下,找出 {要找的東西,例:Kafka topic 命名規則的定義位置}。
動機:{主線為什麼需要,例:接下來要依此規則審查一份新腳本}。

已知線索:{關鍵字、可能的檔名模式、相關目錄;沒有就寫「無」}。

驗收條件:
1. 給出定義該內容的檔案路徑與行號。
2. 若有多處定義,全部列出並指出哪一處最權威(例如 index.md 有引用的那份)。
3. 找不到就明說「找不到」,並列出已搜過的目錄與用過的關鍵字。

回報格式:≤20 行。每個結果一行「檔案路徑:行號 — 一句話說明」。不要貼原文段落。
```

## 2. 實作型(寫新檔/新腳本/新功能)

建議:`general-purpose`,model: sonnet

```
目標:建立 {檔案路徑},內容是 {做什麼}。
動機:{這個產出會被誰/什麼流程使用,例:weak-model session 會照此檔規則派發 subagent}。

規格:
- {規格逐條列出:輸入、輸出、格式、必含章節、風格要求}
- 遵循 {參考檔案路徑} 的既有慣例(命名/格式/語言)。
[- 禁止事項:{不要做的事,例:不要修改規格以外的檔案}]

驗收條件:
1. {可客觀檢查的條件 1,例:檔案存在且包含 X/Y/Z 三個章節}
2. {條件 2,例:所有範例可直接執行,無佔位符殘留}
3. 沒有超出規格的額外內容(不加沒要求的功能/檔案)。

回報格式:≤10 行。回報「已建立 {路徑}+每條驗收條件的自檢結果」。產物存檔,不要貼全文。
```

## 3. 重構型(改既有檔/批次套用模式)

建議:`general-purpose`,model: sonnet(模式已完全定案的機械套用可用 haiku)

```
目標:對 {目標檔案清單或 glob pattern} 套用以下修改:{修改規則,越機械越好}。
動機:{為什麼改,例:規則 X 已更新,舊寫法會誤導}。

修改規則(逐條,含前後對照):
- 把 {舊模式} 改為 {新模式}。範例:改前 `{...}` → 改後 `{...}`
- {規則 2}

邊界:
- 只改上述模式,**不順手修改**其他內容(错字、格式也不動)。
- 遇到不確定是否該改的位置:跳過並在回報中列出,不要自行判斷。
[- 改前先把每個被改的檔備份到 {backup 路徑}。]

驗收條件:
1. 全部目標檔中,舊模式 grep 命中數為 0(或列出合理殘留)。
2. 新模式在每個修改點正確套用。
3. 未列入規則的內容 diff 為零。

回報格式:≤20 行。「改了 N 檔/M 處+跳過清單(檔案:行號+原因)+舊模式殘留 grep 結果」。
```

## 4. 研究型(查證事實/調研方案/讀文件回答問題)

建議:`claude-code-guide`(查 Claude 官方文件)或 `general-purpose`(其他),model: haiku;需要比較與取捨時 sonnet

```
目標:查證/回答:{問題,越具體越好,例:.claude/agents frontmatter 支援哪些欄位與合法值}。
動機:{答案會怎麼被使用,例:要寫進調度守則,錯了會讓後續 session 全跟著錯}。

要求:
- 每個關鍵事實附來源(官方文件 URL 或 檔案路徑:行號)。
- 區分「查證到的事實」與「你的推測」;推測要明標。
- 查不到就寫「查不到:{原因}」,不要用訓練記憶補位。
[- 只查以下來源:{來源白名單};來源之外的不採信。]

驗收條件:
1. 每個問題都有「答案+來源」或「查不到+原因」。
2. 無任何沒有來源的斷言。

回報格式:≤40 行。每個問題一段:「答案 → 來源」。
```

## 5. 審查型(對抗審查/第二意見/read-back 驗證)

建議:`general-purpose`;read-back 用 haiku,對抗審查用 sonnet(高風險用 opus)

```
角色:你是獨立審查者,沒有參與撰寫,你的價值在於找出問題,不是背書。
目標:審查 {檔案路徑清單},判斷是否符合下列標準。
[動機:{產出將用於什麼場合,幫你權衡嚴重度}。]

審查標準(逐條核對):
1. {標準 1,例:規則之間無互相矛盾}
2. {標準 2,例:所有檔案路徑/工具名/模型 ID 真實存在(用工具實際驗證,不要目測)}
3. {標準 3,例:對 Sonnet 等級模型無歧義——每條規則都能明確判斷「做/不做」}

規則:
- 每個 finding 附「檔案:行號 + 問題描述 + 具體修法建議」。
- 按嚴重度排序:會導致錯誤行為 > 歧義 > 風格。
- 沒問題的標準也要逐條回報「通過」,不能整體帶過。
- 不確定的 finding 標「存疑」,不要因為不確定就不報。

回報格式:≤40 行。「逐條標準:通過/finding 清單」+最後一行總判定(可用/需修)。
```

---

## 附:升級重派時的追加段落

任何範本重派升級時(見 model-dispatch.md §5),在 prompt 開頭加上:

```
前情:此任務先前由 {模型} 執行 {N} 次未過驗收。
第 {i} 次交辦內容:{原 prompt 摘要}
第 {i} 次實際輸出:{摘要}
未通過的驗收條件:{哪一條、差在哪}
請先診斷前次失敗的原因,再給出你的做法(如果你打算用同樣方法,說明為什麼這次會不同)。
```
