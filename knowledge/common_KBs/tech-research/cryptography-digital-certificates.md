# 密碼學基礎：Hash、對稱/非對稱加密、數字簽名與憑證

**日期**：2026-06-27
**關鍵字**：密碼學, Hash, 對稱加密, 非對稱加密, RSA, AES, 數字簽名, 數字證書, CA, PKI

## 問題背景

HTTPS、JWT、OAuth、SSH 金鑰等常用技術都建立在密碼學基礎之上。理解 Hash、對稱/非對稱加密、數字簽名的運作原理，有助於正確設計安全機制。

---

## 研究結論

### 一、雜湊（Hash）

雜湊函數將任意長度輸入轉換為固定長度輸出，具備以下特性：

| 特性 | 說明 |
|------|------|
| **Avalanche Effect（雪崩效應）** | 輸入微小改變 → 輸出大幅變化 |
| **Preimage Resistance（不可逆）** | 無法從輸出推導輸入 |
| **Second-preimage Resistance** | 給定 X，難以找 Y 使 hash(X) = hash(Y) |
| **Collision Resistance（抗碰撞）** | 難以找到兩個不同輸入產生相同輸出 |

**常見用途：** 密碼存儲（加 salt）、資料完整性驗證、數字簽名

---

### 二、對稱加密（Symmetric Encryption）

加密與解密使用**同一個金鑰**。

- **優點**：速度快，適合大量資料加密
- **缺點**：金鑰傳輸本身存在安全風險

**常見演算法：**

| 演算法 | 說明 |
|--------|------|
| AES | 目前最廣泛使用，128/192/256 位元金鑰 |
| DES / 3DES | 舊標準，DES 已不安全，3DES 過渡用 |
| Blowfish | 開源，彈性金鑰長度 |

---

### 三、非對稱加密（Asymmetric Encryption）

加密與解密使用**不同金鑰**（公鑰 + 私鑰）。

- **公鑰（Public Key）**：可公開分享，用於加密或驗證簽名
- **私鑰（Private Key）**：秘密保存，用於解密或簽名

- **優點**：不需要預先共享金鑰，解決對稱加密的傳輸問題
- **缺點**：計算量大，速度慢（通常用於交換對稱金鑰）

**常見演算法：** RSA, ECC, DSA, DH（Diffie-Hellman）

> **TLS 交握時的常見模式：** 用非對稱加密安全交換一個對稱金鑰，後續通訊改用對稱加密（兼顧安全與速度）。

---

### 三點五、對稱 vs 非對稱加密的適用選擇

| 維度 | 對稱加密 | 非對稱加密 |
|------|---------|-----------|
| 速度 | 快 | 慢（運算量大） |
| 金鑰管理 | 需安全傳輸金鑰給對方 | 公鑰可公開，私鑰自行保存 |
| 適用場景 | 大量資料加密（資料傳輸主體） | 金鑰交換、數字簽名、少量資料 |
| 常見實作 | AES | RSA、ECC |

> **實務常見模式（TLS / HTTPS）**：用非對稱加密安全交換一把對稱金鑰，後續通訊改用對稱加密，兼顧安全與效能。

---

### 四、數字簽名（Digital Signature）

利用非對稱加密，**私鑰簽名，公鑰驗證**，用於確認資料來源與完整性。

#### 訊息摘要（Message Digest）的角色

簽名前不直接對原文加密，而是先用雜湊函數（Hash Function）將原文轉成**訊息摘要**：

- 訊息摘要長度遠小於原文 → 加解密速度快
- Hash 的 1 對 1 對應特性 → 可確保資料完整性
- Hash 的不可逆性 → 無法從摘要反推原文
- 常見雜湊函數：MD5、SHA-256（MD5 已不推薦，SHA-256 以上為現行標準）

#### 簽名流程

```
發送者 A：
  原文(A) → Hash → 訊息摘要(B)
  訊息摘要(B) + 私鑰加密 → 摘要密文("B")
  發送：明文/密文("A") + 摘要密文("B")

接收者 B：
  用 A 的公鑰解密摘要密文("B") → 取得摘要(B)
  對收到的原文(A) 重新 Hash → 摘要(B')
  比較摘要(B) == 摘要(B') → 驗證未篡改且來自 A
```

#### 加密 + 簽名：雙重保護流程

當同時需要**機密性**（防竊聽）與**來源驗證**（防假冒）：

```
傳送方（持有自己的私鑰 + 接收方的公鑰）：
  Step 1：用接收方的公鑰加密原文 → 密文（保護內容，只有接收方可解）
  Step 2：用自己的私鑰對原文 Hash 後加密 → 摘要密文（簽名，證明來源）
  傳送：密文 + 摘要密文

接收方（持有自己的私鑰 + 傳送方的公鑰）：
  Step 1：用自己的私鑰解密密文 → 取得原文（驗機密性）
  Step 2：用傳送方的公鑰解密摘要密文 → 取得原始摘要
  Step 3：對取得的原文重新 Hash → 新摘要
  Step 4：比對兩個摘要是否相同 → 驗來源與完整性
```

> 防止假冒場景：即使攻擊者取得接收方公鑰，也無法偽造「持有傳送方私鑰」的簽名，接收方比對摘要時必然不符。

**用途：** JWT 簽名、程式碼簽署、Git commit 簽名、合約電子簽章

### 五、數字證書（Digital Certificate / X.509）

數字證書由**受信任的憑證頒發機構（CA）** 簽發，解決「如何信任公鑰真的屬於某人」的問題。

#### 憑證包含

- 證書序列號
- 證書簽名演算法
- 頒發者（CA 名稱）
- 有效期
- 公開金鑰（持有者的）
- CA 的數字簽名（保證憑證未被篡改）

#### 信任鏈（Chain of Trust）

```
Root CA（根憑證，預置在 OS/瀏覽器）
  └─ Intermediate CA（中間憑證）
       └─ End-Entity Certificate（網站憑證，如 *.example.com）
```

#### HTTPS 憑證驗證流程

```
1. 瀏覽器連到 HTTPS 網站
2. 伺服器回傳憑證鏈
3. 瀏覽器驗證：
   - 憑證未過期
   - CA 簽名有效（用 CA 公鑰驗）
   - 域名與 CN/SAN 匹配
4. 驗證通過 → TLS 交握繼續
```

### 六、JWT（JSON Web Token）的密碼學基礎

JWT 本質是 Base64 編碼的三段式結構：

```
Header.Payload.Signature

Header:   {"alg": "RS256", "typ": "JWT"}
Payload:  {"sub": "userId", "exp": 1234567}
Signature: RSA_Sign(Header + "." + Payload, 私鑰)
```

- **HS256**（對稱）：適合單服務，速度快
- **RS256**（非對稱）：適合多服務，Auth Server 持有私鑰，其他服務用公鑰驗證
