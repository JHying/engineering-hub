# 系統性能指標：RPS / QPS / TPS

**日期**：2026-06-27
**關鍵字**：RPS, QPS, TPS, 併發數, Concurrency, Thread, 吞吐量, 高併發

## 問題背景

系統設計與 Code Review 時需要一套共同語言來描述「系統能承受多少量」。RPS / QPS / TPS 是最常見的三個衡量維度，但各自服務不同的系統層級，混用容易造成誤判。

---

## 研究結論

### 三大指標定義

#### RPS（Requests Per Second）— 應用層

每秒請求數，衡量面向用戶的應用層在單位時間內處理的請求量。

```
RPS = 總請求數 / 完成總時間（秒）
```

- 適用：API 服務、Web 應用
- 範例：系統 10 秒內處理 5,000 請求 → RPS = 500
- 注意：高 RPS 時，資料庫層易成為瓶頸，需配合 QPS 同步評估

#### QPS（Queries Per Second）— 資料庫層

每秒查詢數，專門衡量資料庫在高並發讀寫下的性能表現。

```
QPS = 總查詢數 / 完成總時間（秒）
```

- 適用：資料庫效能評估、Index 優化判斷
- 範例：資料庫 20 秒內處理 100,000 次查詢 → QPS = 5,000
- 與 RPS 的差異：QPS 聚焦資料庫層，RPS 聚焦整體應用層

#### TPS（Transactions Per Second）— 交易層

每秒交易數，強調交易的完整性與一致性，常用於金融、支付類系統。

```
TPS = 總交易數 / 完成總時間（秒）
```

- 適用：支付系統、需要保證 ACID 的場景
- 範例：支付系統 30 秒內完成 600 筆交易 → TPS = 20
- 特性：涉及多步驟系統交互，對穩定性與可靠性要求更高

---

### 併發數（Concurrency）

系統在某時間點同時處理的請求數量。

```
併發數 = QPS × 平均回應時間（RT）
QPS    = 併發數 / 平均回應時間（RT）
```

**概念區分**：
- **併發**：同一資源輪流處理多個隊列（單窗口多隊列）
- **並行**：多資源同時處理（多窗口）

---

### Thread 最佳數量公式

```
最佳 Thread 數 = ((Thread 等待時間 + CPU 時間) / CPU 時間) × CPU 數量
```

**範例**：4 核 CPU，CPU 執行時間 20ms，I/O 等待時間 80ms
```
= (100 / 20) × 4 = 20 個 Thread
```

> 超過最佳值後，Thread 競爭加劇導致 QPS 下降。適用於調整 Tomcat / Undertow Worker Thread 上限。

---

### 三大指標間的關係

| 指標組合 | 關係 |
|---------|------|
| QPS vs RT（回應時間） | 負相關：RT 減少 → QPS 增加 |
| 併發數 vs RT | 正相關：RT 增加 → 併發數增加 |
| 高 I/O 負載 | QPS 與 RT 負相關更顯著 |

---

### 資料庫擴展的五大挑戰

1. **資料一致性**：多節點同步困難
2. **分片複雜性**：分片鍵選擇、跨片查詢、分散交易
3. **寫入瓶頸**：無法單靠硬體擴展解決
4. **Lock 競爭**：高並發下易引發死鎖
5. **災難恢復**：故障時的一致性保障

---

### 優化策略

| 層次 | 策略 |
|------|------|
| 應用層 | 讀寫分離、非同步處理、限流控制 |
| 資料層 | 分片、快取（Redis / Caffeine）、NoSQL、CDN、自動擴展 |
| 架構層 | 負載均衡、分散式設計、資源隔離 |

---

### 高併發量級參考（相對標準）

| 規模 | RPS 範圍 |
|------|---------|
| 小型 | 100 – 1,000 RPS |
| 中型 | 1,000 – 10,000 RPS |
| 大型 | 10,000+ RPS（購物節、大型活動） |

> 無絕對標準，高併發定義取決於系統設計能力、用戶基數、應用類型與基礎設施。
