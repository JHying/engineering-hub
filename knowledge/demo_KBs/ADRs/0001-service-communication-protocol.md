# ADR-0001 服務間通訊協議選型

| 欄位 | 內容 |
|------|------|
| 狀態 | Accepted |
| 日期 | 2024-01 |
| 決策者 | 架構團隊 |

---

## 情境

訂單平台共有 4 個微服務。初期討論了三種服務間通訊方式：REST、gRPC、Kafka。
需要決定各場景下採用哪種協議，避免混用造成維護困難。

## 問題

不同通訊需求（同步 vs 非同步、內部 vs 外部）應如何選擇協議？

## 決策

採用「場景對應協議」原則：

| 場景 | 協議 | 理由 |
|------|------|------|
| 外部請求入口 | REST（HTTP） | 對外 API 標準，客戶端相容性最佳 |
| 內部同步呼叫 | gRPC | 強型別契約、低延遲，適合服務間強依賴 |
| 跨服務非同步事件 | Kafka | 解耦生產者 / 消費者，天然支援重試與回放 |

## 後果

**正面：**
- 各協議職責清晰，新人容易判斷「這個呼叫該用什麼」
- gRPC 的 .proto 作為服務間契約，變更有明確的 breaking change 判斷依據

**負面 / 注意事項：**
- 需維護 .proto 檔案並管理版本相容性
- Kafka consumer 需實作冪等保護（防重複消費）

## 替代方案（被放棄）

- **全用 REST**：缺乏型別保護，服務間耦合高，回傳格式容易漂移
- **全用 Kafka**：同步場景（需即時回應）不適合 Kafka，會過度複雜化
