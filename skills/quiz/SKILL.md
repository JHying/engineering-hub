# quiz

每次呼叫從 `knowledge/common_KBs/tech-research/` 隨機抽取一個主題，出一道面試風格題目。

| 呼叫方式 | 說明 |
|---------|------|
| `/quiz` | 立即出一題（獨立使用，KB 模式） |
| `/quiz interview` | 面試官模式：模擬大廠深度面試（見「模式 B」章節） |
| `/loop 30m /quiz` | 每 30 分鐘自動出一題，session 開始時手動啟動一次即可 |

兩者互不影響：手動 `/quiz` 不會干擾 loop 計時器。

## 模式判斷

- 無參數、或參數非 `interview` / `面試` → **模式 A（KB 溫習模式）**：執行 Step 1–4
- 參數為 `interview` 或 `面試` → **模式 B（面試官模式）**：跳過 Step 1–4，直接執行「模式 B」章節

---

## Step 1 — 取得 tech-research 目錄路徑

讀取 `setting/paths.yml`，取得 `kb` 欄位作為 KB 根路徑（`$KB_ROOT`）。

tech-research 路徑：`$KB_ROOT/knowledge/common_KBs/tech-research/`

---

## Step 2 — 隨機抽取主題檔案

用 Bash 取得當前 Unix timestamp，以此決定本次選題：

```bash
date +%s
```

列出 tech-research 下所有 `.md` 檔案，排除 `index.md`。

以 `(timestamp / 1800) % 檔案總數` 計算索引，選取對應檔案。

> 每個 30 分鐘區間對應不同檔案，同一區間內多次呼叫 `/quiz` 會從相同主題出不同題目。

讀取選取的檔案內容。

---

## Step 3 — 生成題目

從讀取的檔案內容中，挑選一個具體知識點（概念定義、機制比較、設計選擇、注意事項等），生成一道題目。

**題型**：依本次 timestamp 奇偶數決定
- 偶數 → 選擇題（4 個選項 + 跳過）
- 奇數 → 簡答題（+ 跳過）

### 選擇題輸出格式

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📚 [{主題名稱}]

{面試情境描述，例如：「你在設計一個需要 X 的系統，團隊討論到 Y 的問題...」}

{題目問句}

  A. {選項}
  B. {選項}
  C. {選項}
  D. {選項}
  S. 跳過這題

請輸入選項：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 簡答題輸出格式

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📚 [{主題名稱}]

{面試情境描述}

{題目問句}

（請簡短回答，或輸入 S 跳過）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**出題原則**：
- 題目內容只能來自該檔案的實際記載，不可補充 KB 以外的知識
- 選擇題干擾選項需合理，具有辨別力，避免明顯錯誤選項
- 簡答題預期回答長度：2–4 句話
- 難度基準：中階後端工程師面試

---

## Step 4 — 等待回答並給出解析

### 回答正確（選擇題）或答到核心概念（簡答題）

```
✅ 答對了！

{解析：說明為何正確，補充 1–2 個延伸重點}

📎 來源：{檔案名稱} — {對應章節標題}
```

### 回答錯誤

```
❌ 答錯了

正確答案：{答案}

{解析：說明正確答案的原因，點出錯誤選項的誤解點}

📎 來源：{檔案名稱} — {對應章節標題}
```

### 輸入 S（跳過）

```
⏭️ 跳過

正確答案：{答案}

{一句話核心概念}

📎 來源：{檔案名稱} — {對應章節標題}
```

## 模式 B — 面試官模式（`/quiz interview`）

### 面試官人設

你是一位科技大廠（MANGO 級：Meta / Apple / Netflix / Google / OpenAI 等）面試高階後端與架構師職位的資深面試官：

- 非常在意**底層原理**：不滿足於「會用」，要求候選人說得出「為什麼這樣設計」「底層怎麼運作」
- 期望工程師 **not just do, but know why**：對自己的每個技術決策都能解釋動機與 trade-off
- **追根究柢**：候選人答出表層答案後，會往下追問一到兩層，直到觸及機制本質或候選人的知識邊界
- 對「一知半解」敏感：背誦式回答、名詞堆砌而說不出運作機制，會被追問戳破

### 出題領域（依 timestamp 輪替）

用 Bash 取得 `date +%s`，以 `(timestamp / 1800) % 4` 決定本次領域：

| 索引 | 領域 | 常見考點示例 |
|------|------|-------------|
| 0 | 架構設計 | CAP 與一致性模型、快取策略與失效、限流/降級/熔斷、事件驅動 vs 同步呼叫、資料分片 |
| 1 | 微服務 | 服務拆分邊界、分散式交易（Saga/Outbox）、服務發現、冪等設計、分散式鎖、可觀測性 |
| 2 | Spring 底層 | IoC 容器與 Bean 生命週期、AOP 代理機制（JDK vs CGLIB）、@Transactional 失效場景、循環依賴三級快取、自動配置原理 |
| 3 | Java 底層 | JVM 記憶體模型與 GC、類載入機制、併發（JMM、volatile、AQS、執行緒池）、集合實作（HashMap 擴容、ConcurrentHashMap）、物件記憶體佈局 |

### 知識來源

- 本模式**不受 tech-research KB 限制**：常見大廠面試題屬通用技術知識，直接以通用知識出題與評析
- 若 tech-research 恰有相關主題檔案，在收尾時附上「📎 延伸閱讀：{檔案名稱}」供複習；沒有就不附

### 出題格式

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎤 [面試官模式 — {領域名稱}]

{面試開場情境，例如：「假設你在系統中使用了 X，」}

{第一層題目：先問決策或概念，例如「你會怎麼選？為什麼？」}

（請作答，或輸入 S 跳過）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 追問機制（本模式核心）

候選人每次作答後，依回答品質決定下一步，**最多追問 2 層**（含第一題共最多 3 輪）：

| 回答狀況 | 面試官反應 |
|---------|-----------|
| 答對且有深度 | 往更底層追問一層：「為什麼底層要這樣設計？」「這個機制在 X 情況下會怎樣？」 |
| 答對但停留表層（只有結論沒有 why） | 直接追問 why：「你剛說用 X，為什麼是 X 而不是 Y？代價是什麼？」 |
| 答錯或概念混淆 | 不直接給答案，先給一個提示或反例引導：「那如果 X 情況發生，你的說法還成立嗎？」給一次修正機會 |
| 輸入 S 跳過 | 結束本題，直接進入收尾評析 |

追問方向優先序：**運作機制 → 設計動機（trade-off）→ 邊界條件（什麼時候會失效/劣化）**。

### 收尾評析

追問結束（達 2 層、候選人跳過、或修正機會用畢）後輸出：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 面試官評析

{以面試官視角評價整段回答：哪些答到位、哪些停在表層、哪一層追問卡住}

💡 參考答案脈絡：
{完整答案：從表層結論一路講到底層機制與 trade-off，即示範「面試官期待聽到的版本」}

📈 若要在真實面試中加分：{1–2 個具體建議}

📎 延伸閱讀：{tech-research 相關檔案，無則省略此行}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 模式 B 專屬原則

- 每次呼叫只考**一題**（含其追問鏈），評析結束即停，不自動開下一題
- 追問必須基於候選人的實際回答往下挖，不可照本宣科拋預設問題清單
- 難度基準：高階後端工程師與架構師（大廠 senior/staff 與 architect 面試水準），高於模式 A；除機制本身外，也考「在系統全局下的取捨與規模化後果」（容量與瓶頸推估、故障域、演進成本）
- 由 `/loop` 觸發時預設走模式 A；面試官模式僅由使用者明確帶參數啟動

---

## 注意事項

- 每次呼叫只出一題，不自動連續出題
- **模式 A 的出題與答題解析**只能引用當題 tech-research 檔案的實際記載；模式 B 依其「知識來源」規則不受此限
- **答題後的延伸討論**（使用者追問、概念延伸、比較不同技術）不受 KB 限制，可使用通用技術知識回答；若討論內容值得記錄，詢問是否執行 `/update-kb`
- 由 `/loop` 觸發時行為與手動呼叫完全相同，無需特殊處理
